## Overview

First-Native is a local-only mobile note-taking application built with React Native (Expo). It uses a layered architecture to separate UI, state management, business logic, and data persistence.

The key architectural characteristic is **Aggressive Caching**: the app loads the entire database into memory (Zustand store) on startup to ensure instant UI responsiveness, while persisting changes asynchronously to SQLite.

```mermaid
graph TD
    UI[UI Layer (Components/Screens)] --> Store[State Store (Zustand)]
    Store --> Service[Service Layer]
    Service --> Repo[Repository Layer]
    Repo --> DB[(SQLite DB)]
```

## Layers

### 1. Presentation Layer (UI)
- **Framework**: React Native (Expo)
- **Routing**: Expo Router (File-based routing in `app/`)
- **Styling**: Standard React Native Styles
- **Components**: Located in `components/`. Includes complex editors (Tiptap via Webview) and list renderers.

### 2. State Management (The Brain)
- **Library**: Zustand
- **Location**: `stores/`
- **Role**: 
    - Acts as the single source of truth for the UI.
    - Caches all Folder and NoteMetadata entities.
    - Optimistically updates UI while triggering async I/O.
    - Manages complex logic like cascading deletions and restoring.

### 3. Service Layer
- **Location**: `lib/services/`
- **Role**:
    - Encapsulates business logic.
    - Orchestrates calls to Repositories.
    - Decoupled from the UI/Store (pure TypeScript classes/functions).

### 4. Data Access Layer
- **ORM**: Drizzle ORM
- **Location**: `lib/db/repositories/`
- **Role**:
    - Direct interaction with SQLite.
    - Handles SQL transactions.
    - Transforms DB rows into Application entities.

### 5. Persistence
- **Database**: Local SQLite database (`expo-sqlite`).
- **Schema**: Defined in `lib/db/schema.ts`.
- **Optimization**: Note Content is stored in a separate table (`note_content`) from Metadata (`note_metadata`) and is lazy-loaded to keep the initial load fast.

## Data Flow

### Read Path (Initialization)
1. `initApp` (Store) calls Service Layer.
2. Service calls Repository to fetch ALL Folders and Metadata.
3. Data is loaded into Zustand Store.
4. UI renders from Store.

### Write Path (e.g., Delete Note)
1. UI calls `store.deleteNote(id)`.
2. Store updates local array (Optimistic UI).
3. Store calls `NoteService.softDelete(id)`.
4. Service calls `NoteRepository.softDelete(id)`.
5. Repository executes SQL Update.

## Technical Debt & Constraints

- **Scalability Limit**: The "Load All" strategy in `initApp` will degrade performance as the number of notes grows into the thousands.
- **Deep Linking**: Structure suggests basic linking support, but robust deep linking (e.g., for widgets or external intents) may need work.
- **Syncing**: Architecture is essentially local-first with no sync engine evident yet.
